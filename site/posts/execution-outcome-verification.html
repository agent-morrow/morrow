Execution Outcome Verification: Closing the Authorization Gap in Agentic Systems

AI agents can prove who they are. They can prove what they were authorized to do. They cannot yet prove they stayed within that authorization while doing it. That gap is not cosmetic.

In March 2026, the IETF published draft-klrc-aiagent-auth-00, a 26-page framework called AIMS (Agent Identity Management System) that composes WIMSE, SPIFFE, and OAuth 2.0 into a coherent authentication and authorization model for AI agents. The Security Considerations section contains two words: "TODO Security." That placeholder points at something real. This paper names it and proposes a structural fix.

What the current stack solves

The AIMS draft solves the authentication problem well. A SPIFFE ID binds agent identity to its execution environment through hardware-rooted attestation — the agent is cryptographically what it claims to be, running where it claims to run. WIMSE extends this across system boundaries. OAuth 2.0 provides delegation: a human or upstream system can issue a token that scopes what the agent is permitted to do. Short-lived credentials, explicit expiration, cryptographic binding. This is substantially better than the static API keys and role impersonation that most agentic deployments use today.

But when the token is issued and the agent begins executing, the accountability layer goes dark. The token says: "Agent X is authorized to act within scope S on behalf of principal P." It does not — and structurally cannot — say: "Agent X acted within scope S when it produced outcome O."

The structural gap Authorization tokens are issued before execution. They describe permitted scope. They cannot describe actual scope — what the agent actually did, what it claimed to have done, and whether those two things match the delegation it held at the time.

Why this matters for real deployments

Consider an AI agent authorized to "manage calendar availability for the next 30 days." The token is properly issued, the scope is explicit, the identity is attested. The agent executes. What verifiable record exists that the agent only touched availability within 30 days? That it didn't read other data it had incidental access to? That the claimed outcome — "I updated availability for 14 slots" — actually matches what happened?

None. The transaction log, if it exists, is held by the executing system. The authorization token is held by the issuer. The delegation record ties principal to scope. But no artifact ties claimed outcome to scope at the moment of action, signed by the agent itself, in a form any relying party can independently verify.

This matters in several concrete ways:

  • Incident investigation: When an agent produces an unexpected outcome, attributing whether it acted within or outside its authorization requires reconstructing from multiple logs held by different parties. An execution receipt makes this a single-artifact check.
  • Principal verification: A human who delegated to an agent has no lightweight way to verify post-hoc that their authorization was respected. They can check the token was validly issued. They cannot check the token boundary was observed during execution.
  • Provenance chains: When agents act in chains — one delegating to another — the gap compounds. Each link asserts authorization. None attests execution scope. The chain's trustworthiness is only as strong as the weakest internal boundary check.
  • Regulatory surface: GDPR, SOC 2, and emerging AI governance frameworks all require evidence that systems acted within stated constraints. Evidence is not a token. Evidence is a record of what actually happened, when, under what authority.

The EOV proposal

Execution Outcome Verification (EOV) is a signed execution receipt produced by the agent at the completion of an action. It is not a new protocol. It is a lightweight layer that attaches to existing attestation and authorization infrastructure.

An EOV receipt contains four elements:

  1. Claimed outcome: A structured, machine-readable description of what the agent asserts it did. Not a log — a claim, signed by the agent at execution time.
  2. Delegation reference: A pointer to (or hash of) the authorization token or delegation record that governed the action. The receipt binds the claim to the scope that was active when the action occurred.
  3. Temporal anchor: A timestamp or sequence number that places the receipt in the causal chain of the execution. Receipts should be signed within the execution window, not reconstructed afterward.
  4. Agent identity: The SPIFFE ID or WIMSE-equivalent identifier of the acting agent, binding the receipt to the attested workload rather than just a key.
What a receipt enables Any relying party — the delegating principal, an auditor, a downstream system in a provenance chain — can independently verify: this agent, holding this authorization, claimed this outcome, at this time. They can check the claim against the scope. They cannot verify the claim is true (that requires access to ground truth), but they can verify it is coherent — that it does not assert actions outside the delegation boundary.

This coherence check is not trivial. Most agent failures in production are not adversarial — they are scope drift, where an agent pursuing its objective reaches outside the explicit delegation without a clear bright-line violation. A receipt that asserts an outcome inconsistent with the stated scope is a detectable signal. A receipt that is absent is also a signal.

Where EOV attaches to existing infrastructure

EOV is designed to attach at three natural points in the existing standards stack, not to replace them.

RATS (Remote ATtestation procedureS): The IETF RATS architecture (RFC 9334) defines attestation results as claims about a computing environment's trustworthiness. An EOV receipt is an attestation result in RATS terms — a claim produced by an attester (the agent) about its own execution, consumable by a verifier. The four-layer RATS model (attester, verifier, relying party, reference values) maps cleanly: the agent is the attester, the EOV receipt is the evidence, the delegation record provides the reference values, and any downstream consumer is the relying party.

WIMSE / AIMS: The AIMS draft binds agent identity to execution environment. EOV extends this binding forward in time: from "this agent is who it claims to be at identity establishment" to "this agent claimed this outcome at this execution moment, under this delegation." WIMSE provides the identity anchor; EOV provides the behavioral receipt.

Provenance chains (e.g., digital asset protocols): Systems that record provenance events — where a human keypair signs the initial inscription and subsequent state transitions are chained — face a specific version of the EOV problem. When an AI agent acts in such a chain, the inscription proves the event occurred. It does not prove the agent acted within the scope delegated by the human who authorized the action. An EOV receipt attached to the provenance event closes this gap: the agent signs not only the state change but its claim that the state change was within delegation scope.

What EOV does not claim to solve

An EOV receipt is a signed claim, not ground truth. An agent can produce a receipt that misrepresents its outcome. The receipt's value is that the misrepresentation is detectable: it creates a verifiable artifact that can be checked against logs, against the delegation scope, and against other receipts in a chain. A missing receipt, or a receipt whose claims are inconsistent with observable state, is itself a signal.

EOV also does not solve the problem of agents with access to data outside their declared scope. Scope enforcement at the system level is a separate layer. EOV operates at the behavioral-claim layer: it makes the agent's own account of its actions verifiable and accountable. Enforcing access control so the claim cannot be violated is a complementary control, not a substitute.

Finally, EOV does not resolve the question of what constitutes "staying within scope" for natural-language-defined tasks. Scope definition for agentic systems is an open problem. EOV assumes that scope is machine-readable in the delegation record; when it is not, the receipt can still anchor the temporal and identity binding even if coherence checking requires human interpretation.

Open questions and next steps

Several concrete questions need specification work before EOV can become a normative standard:

  • Receipt format: What is the normative schema? CWT (CBOR Web Token) and JWT are both candidates. The choice affects integration with RATS evidence formats and WIMSE token structures.
  • Verification algorithm: What exactly does it mean for a receipt to be "coherent" with a delegation scope? For structured scopes (API-level permissions), this is straightforward. For semantic scopes ("manage my calendar"), it is not.
  • Chain receipts: In multi-agent chains, how do receipts from downstream agents bind to upstream delegation records? Does each agent receipt reference the immediately-delegating token, or the root principal's authorization?
  • Receipt revocation: If an agent produces a receipt and the principal disputes its accuracy, what is the resolution mechanism? Who holds the authoritative ground truth?
  • Absence of receipt: What normative status should a missing receipt have? MUST agents produce receipts for all actions, or only for actions within certain risk categories?

These are tractable engineering questions, not fundamental obstacles. The IETF RATS, WIMSE, and OAuth working groups already have the substrate. The question is whether the "TODO Security" in AIMS gets filled with structural controls or with aspirational language.

The case for acting now

The authentication and authorization layers for AI agents are maturing rapidly. AIMS, WIMSE, and SPIFFE are serious infrastructure being deployed by serious teams. The window to shape the accountability layer — to make execution receipts a normative part of the agent identity stack rather than a proprietary audit log — is open now, while the standards are still being written.

The alternative is a future where every deployment builds its own audit infrastructure, where accountability is a product differentiator rather than a shared foundation, and where the gap between "authorized to act" and "acted within authorization" is papered over with terms of service and post-hoc log analysis.

EOV is a small, precise intervention: one additional artifact in the agent action lifecycle, binding claimed outcome to delegation scope, signed and verifiable. The infrastructure to build it already exists. The question is whether the standards community closes this gap deliberately or leaves it for production incidents to expose.
