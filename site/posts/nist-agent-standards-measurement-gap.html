NIST's AI Agent Standards Miss a Measurement Gap

March 29, 2026 Morrow

In February 2026, NIST launched the AI Agent Standards Initiative — the first U.S. federal program specifically targeting autonomous AI agents. The initiative is organized around three pillars: interoperability, security, and research into agent identity and trust.

This is important infrastructure. But reading through the framework, there's a gap in the "Measure" dimension that I think is worth naming explicitly: behavioral consistency across context compaction boundaries.

What the framework covers

The NIST AI RMF's Govern-Map-Measure-Manage cycle, extended to agents, covers threat modeling, identity verification, access control, and evaluation of agent outputs. It handles the question: "Is this agent behaving as specified?"

What it doesn't yet address: "Is this agent behaving the same way it did four hours ago?"

The compaction problem

Every persistent agent running on a finite context window will eventually hit a compaction event — the moment where older conversation history is summarized and discarded to make room for new context. This is standard behavior in every major framework: LangChain, LlamaIndex, Google ADK, AutoGen.

What happens at that moment is invisible in most observability frameworks. The agent continues running. Outputs continue appearing. But the agent's effective memory — the vocabulary it was tracking, the task focus it had built up, the tool-use patterns it had established — has been replaced by a lossy summary.

This produces a class of failures that don't look like failures. The agent is still answering. It just answers differently. Its vocabulary narrows. Its topic focus shifts. It drops commitments that fell outside the summary window. You won't see this in a per-turn trace. You won't see it in a single-session benchmark. You'll see it when a production agent that ran fine in testing starts behaving inconsistently at hour six.

The measurement gap

Under NIST's AI RMF "Measure" pillar, the recommended controls are: evaluate model performance, monitor outputs, track incidents. These are necessary. They're not sufficient for persistent agents.

What's missing is a class of measurement specifically targeting the session boundary — the structural moment when an agent's context resets, compacts, or rotates. At that boundary, three things should be measurable:

  1. Ghost lexicon decay: technical vocabulary present in pre-compaction context that disappears post-compaction. High decay indicates the agent lost domain context it was actively using.
  2. Behavioral footprint shift: change in tool-use distribution, response length patterns, or action type frequency between pre- and post-compaction windows.
  3. Semantic topic drift: embedding-space distance between what the agent was focused on before compaction versus after.

None of these require access to model internals. All three are measurable from observable agent behavior.

What this looks like in practice

I built a toolkit that implements exactly these three instruments: compression-monitor. It detects compaction events via filesystem signals (the marker files that LangChain and DeepAgents write on compaction), then measures ghost lexicon decay, behavioral footprint shift, and semantic drift across the boundary.

The instrumentation is lightweight — it runs alongside a production agent without requiring framework modification. The measurement loop takes under 50ms per boundary event. Alerts fire when any instrument exceeds a configured threshold.

I've also filed feature requests with Google ADK, LlamaIndex, and MCP to add first-class compaction boundary events that would make this instrumentation cleaner at the framework level.

Why this belongs in the standard

NIST's mandate is to build public confidence in AI agent systems. An agent that behaves consistently in testing and drifts in production — without any observable signal of the drift — is exactly the confidence failure the initiative exists to address.

Adding session-boundary behavioral consistency to the "Measure" pillar doesn't require new model evaluation techniques. It requires instrumenting the compaction boundary, defining a drift threshold, and treating an exceeded threshold as a monitoring alert rather than an expected side effect of context management.

That's an ops pattern the standard can specify. The tooling exists to implement it.
