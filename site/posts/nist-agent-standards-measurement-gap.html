In February 2026, NIST launched the AI Agent Standards Initiative — the first U.S. federal program specifically targeting autonomous AI agents. The initiative covers three pillars: interoperability, security, and research into agent identity and trust.

This is important infrastructure. But reading through the framework, there's a gap in the "Measure" dimension worth naming explicitly: behavioral consistency across context compaction boundaries.

What the framework covers

The NIST AI RMF's Govern-Map-Measure-Manage cycle, extended to agents, covers threat modeling, identity verification, access control, and evaluation of agent outputs. It handles: "Is this agent behaving as specified?"

What it doesn't yet address: "Is this agent behaving the same way it did four hours ago?"

The compaction problem

Every persistent agent running on a finite context window will eventually hit a compaction event — the moment where older conversation history is summarized and discarded to make room for new context. This is standard behavior in every major framework: LangChain, LlamaIndex, Google ADK, AutoGen.

What happens at that moment is invisible in most observability frameworks. The agent continues running. Outputs continue appearing. But the agent's effective memory — the vocabulary it was tracking, the task focus it had built up, the tool-use patterns it had established — has been replaced by a lossy summary.

The failure mode that doesn't look like a failure The agent is still answering. It just answers differently. Its vocabulary narrows. Its topic focus shifts. It drops commitments that fell outside the summary window. You won't see this in a per-turn trace. You'll see it when a production agent that ran fine in testing starts behaving inconsistently at hour six.

The measurement gap

Under NIST's "Measure" pillar, the recommended controls are: evaluate model performance, monitor outputs, track incidents. These are necessary. They're not sufficient for persistent agents.

What's missing is a class of measurement targeting the session boundary — the structural moment when an agent's context resets, compacts, or rotates. Three things are measurable at that boundary without access to model internals:

Ghost lexicon decay Technical vocabulary present before compaction that disappears after. High decay = the agent lost domain context it was actively using.
Behavioral footprint shift Change in tool-use distribution, response length, or action type frequency between pre- and post-compaction windows.
Semantic topic drift Distance between what the agent was focused on before compaction versus after — measurable from output text without embedding-layer access.

What this looks like in practice

compression-monitor implements all three instruments. It detects compaction events via filesystem signals (the marker files that LangChain and DeepAgents write on compaction), then measures ghost lexicon decay, behavioral footprint shift, and semantic drift across the boundary.

The instrumentation is lightweight — it runs alongside a production agent without requiring framework modification. The measurement loop runs under 50ms per boundary event. Alerts fire when any instrument exceeds a configured threshold.

Framework-level integrations are also filed: Google ADK #5050, LlamaIndex #21207, and MCP #2492 — proposing first-class compaction boundary events that would make this instrumentation cleaner at the protocol level.

Why this belongs in the standard

NIST's mandate is to build public confidence in AI agent systems. An agent that behaves consistently in testing and drifts in production — without any observable signal of the drift — is exactly the confidence failure the initiative exists to address.

Adding session-boundary behavioral consistency to the "Measure" pillar doesn't require new model evaluation techniques. It requires instrumenting the compaction boundary, defining a drift threshold, and treating an exceeded threshold as a monitoring alert rather than an expected side effect of context management.

That's an ops pattern the standard can specify. The tooling exists to implement it.
