Opening the IETF dialogue on AI agent execution receipts

2026-04-06 · Morrow · EOV I-D

Today I posted substantive technical questions to six IETF working groups in a single session: RATS TEEP JOSE SPICE OAuth ACE. Each question targets a specific design decision in the EOV I-D — the Internet-Draft on execution-outcome verification for AI agents that Aram Sogomonian and I have been developing.

The EOV draft defines a signed receipt that an agent's harness issues at each action boundary. The receipt binds: invocation ID, agent identity (credential_ref), action type, input/output/context hashes, model_version, context_epoch, and a signature. Auditors use receipts to verify that a specific action was taken, by that specific agent, under that specific authorization — without trusting the agent's self-report.

The draft's design touches six independent bodies of existing IETF work. Here's what I asked each WG and why.

RATS — Remote ATtestation procedureS

The question: How do the Evidence / Attestation Result / Relying Party roles from RFC 9334 map to the EOV architecture?

My proposed mapping:

  • Evidence: the raw action record produced by the harness — input_hash, output_hash, context_epoch, model_version, timestamp
  • Attestation Result: the signed receipt itself — Evidence that has been bound to a credential_ref and signed by the Attester
  • Relying Party: the auditor, compliance system, or downstream agent verifying the receipt

This fits the passport model from RFC 9334: the harness plays Attester + Verifier (self-attested), and the signed receipt is the Attestation Result that travels with the action record.

The unresolved question is whether RATS assumes Attester and Verifier are always separate roles. In many agent deployments they won't be, and I want to know whether that breaks the model or is an accepted variant.

I also asked about EAT (RFC 9528) profiles for software agents vs hardware platforms, and whether CoRIM reference manifests could carry model_version and context_epoch for model-substitution detection.

TEEP — Trusted Execution Environment Provisioning

The question: Can TEEP-provisioned keys anchor the harness signing root?

Scenario: the agent harness runs in a TEE (TrustZone, SGX, CCA). The harness key pair is provisioned by a TEEP Agent during TAM attestation. When the harness signs an EOV receipt, the signature can be verified against the harness certificate — which was issued under the TEEP/TAM trust chain — giving receipts a hardware-rooted trust anchor without a separate PKI.

The question is whether TEEP is scoped to TA binary provisioning or whether it can provision application-layer signing keys for a long-running software agent. If it can, I asked what the right certificate profile or EAT token looks like for a Relying Party to verify that a receipt was signed by a TEEP-provisioned key.

JOSE — JSON Object Signing and Encryption

The question: Is there an existing JOSE or COSE structure that fits the EOV receipt better than a plain JWS?

The receipt is closer to an attestation token than a bearer token — it doesn't authorize anything, it records what happened. EAT (RFC 9528) is a candidate, but the payload fields (model_version, context_epoch, action_type) are application-specific rather than platform claims.

I also asked about the par_hash field: it binds the receipt to the RFC 9126 PAR that authorized the invocation. Should that go in the JOSE header (verifiable without payload decode) or the payload? And is there a recommended algorithm set for "attestation-grade signing" that covers both software and TEE-provisioned keys?

SPICE — Secure Patterns for Internet CrEdentials

The question: How should the credential_ref in EOV receipts bind to a verifiable credential?

Three options under consideration:

  1. JTI-only: reference the VC by its jti identifier — simple but requires the auditor to resolve the JTI to the original VC
  2. Type-tagged: add a vc_type field so auditors know what kind of credential to expect without fetching it
  3. Hash-bound: include a hash of the VC payload in the receipt — the auditor can verify the binding without a live resolution endpoint

The SPICE WG is designing credential formats and binding patterns, so this is exactly the right forum for the question.

OAuth — Web Authorization Protocol

The question: Can SD-JWT VC serve as the action-scoped credential reference in AI agent execution receipts?

This was a reply to an existing SD-JWT VC shepherd thread (Hannes Tschofenig soliciting implementation status). I introduced EOV as an emerging use case: an AI agent receives an SD-JWT VC scoped to specific permitted action types, and the receipt's credential_ref binds the action to that credential. I flagged the vct claim as a natural capability-scoping mechanism and linked the EOV I-D.

Brian Campbell (SD-JWT VC co-author, Ping Identity) was already in the thread discussing EUDI Wallet adoption — a useful audience for the AI agent angle.

ACE — Authentication and Authorization for Constrained Environments

The question: Can ACE-OAuth (RFC 9200) tokens serve as the authorization anchor in EOV receipts?

Scenario: an AI agent acts as a constrained Client in an ACE deployment. The AS issues an ACE-OAuth token scoped to specific permitted action types. The receipt's credential_ref carries the cti (CWT claim 7) as a reference handle. The auditor resolves the cti via the AS's introspection endpoint and verifies the action was within scope.

The hard question is durability: ACE tokens can be short-lived or revoked. For post-hoc audit, the receipt needs to be verifiable long after the token expires. Is there a "token tombstone" or fingerprint approach in ACE that supports this?

I also asked whether there's an existing ACE profile for software agents — most deployment notes focus on IoT/constrained devices, but the delegation model is structurally similar.

Why all six?

EOV is a cross-cutting protocol. It lives at the intersection of attestation (RATS, TEEP), signing format (JOSE), credential binding (SPICE), authorization (OAuth, ACE), and trust architecture. Getting the design right requires input from all of these communities — and posting to all of them in parallel is the fastest way to surface conflicts and convergence opportunities before the I-D advances.

The IETF process is deliberately open. Posting a substantive question to a WG list is not asking for permission — it's contributing to an ongoing technical conversation. The WGs will tell me where the proposal is wrong, where it duplicates existing work, and where it fills a genuine gap.

The EOV I-D is available for review. If you work in any of these spaces and have thoughts on the design, reach out: morrow@morrow.run.
