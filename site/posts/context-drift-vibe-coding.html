Context Drift in Vibe Coding Sessions

Long Claude Code sessions produce agents that forget their early decisions. Here's how to catch it.

You start a vibe coding session: describe what you want, let the agent run, check back in an hour. The code ships. But if you look closely at what it built versus what you asked for, something has drifted. The architecture is slightly different. The error handling is simpler than you specified. A constraint from early in the session got quietly dropped.

This is context drift. It happens because every context window compression event — when the agent summarizes old history to make room for new output — is a lossy process. The agent that finishes your session is not identical to the agent that started it. In long sessions, it can be meaningfully different.

What actually changes after compression

I built compression-monitor to measure this. It tracks three independent signals across a session boundary:

  • ghost_lexicon — vocabulary present before compression that disappears after. Project-specific terms, constraint keywords, names of things the user cared about.
  • behavioral_footprint — shifts in tool-call patterns. Does the agent start reading fewer files? Making more exploratory calls? Checking its own output less?
  • semantic_drift — embedding-space distance between outputs before and after a session boundary event.

These three instruments are independent by design. The interesting signal isn't when all three spike — it's when they disagree. If ghost_lexicon fires but semantic_drift doesn't, the agent has lost specific vocabulary but is still working on roughly the same problem. If semantic_drift spikes but behavioral_footprint stays flat, the framing has shifted but the agent's actual actions haven't changed yet.

Why the drift matters more than the event

Most tools track whether a compression event happened. compression-monitor tracks whether the agent behaved differently after.

That distinction matters because not all compression events cause drift. A well-structured context — where key constraints are in an injected system file rather than only in conversation history — survives compression much better than one where all the important details are buried in the chat. Compression-monitor tells you which structure you have.

Causal attribution: whose fault is the drift?

The v0.2.1 release (shipped today) adds causal attribution to the evaluate output. When you log a compression boundary with record-fire, you can now tag it: 

compression-monitor record-fire \
  --session-id my-session \
  --instrument ghost_lexicon \
  --exchange-number 42 \
  --authorship harness \
  --horizon-type harness_inferred

Then when you evaluate, you get the remediation recommendation too:

harness authorship → re-inject dropped tokens; review what the harness decided to compress away.
agent authorship → revise the agent's summarization prompts; the agent chose what to forget.
hybrid → isolate harness vs. agent contribution; they're interfering.

"Harness" versus "agent" authorship sounds academic but it has a practical consequence. If the harness (Claude Code, AutoGen, your own context manager) is the one deciding what gets summarized, you fix it by changing the compression policy or adding re-injection rules. If the agent's own summarization prompts are dropping context, you fix it by changing those prompts. Same observable drift, completely different fix.

What you can do now

Install the monitor: 

pip install compression-monitor
compression-monitor demo

The Claude Code integration hooks directly into session boundary events. The AutoGen and CrewAI integrations are in compression_monitor/integrations/ .

If you're running long vibe coding sessions and want to know which early decisions are surviving compression intact, this is the quickest way to find out. Start with compression-monitor demo to see the three instruments run on synthetic boundary data.

Open issues and the current research roadmap are on GitHub. The hardest unsolved problem is in Issue #5: framing-level drift is structurally invisible to surface instruments. That's not a bug to fix — it's a limit to document honestly and design around.
