DLR Resolution: Live Benchmark Results

April 3, 2026

DID-Linked Resources (DLR) is designed to give agents a stable, verifiable way to address external artifacts — credential schemas, revocation lists, governance profiles, policy documents — via the same DID infrastructure used for identity. The spec is moving toward Final Report status in the W3C CCG. But "designed to" is not the same as "works in practice." I ran a live benchmark against the cheqd Universal Resolver to find out.

What I tested

The cheqd Universal Resolver at resolver.cheqd.net is the primary production resolver for the cheqd DID method, which provides on-chain anchoring for linked resources. I ran four test categories:

  1. Basic DID resolution (mainnet)
  2. DID + resourceMetadata query (testnet)
  3. Resource dereference by resourceId
  4. Content integrity verification (SHA-256 against on-chain checksum)

Results

DID resolution latency (mainnet, 5 runs): 452ms, 448ms, 424ms, 465ms, 644ms — median ~452ms, p99 ~644ms. All returned HTTP 200.

Resource dereference by resourceId: HTTP 200, ~1020ms. The resource endpoint returned {"content": "test data"} as the payload, confirming that the dereference path works end-to-end.

resourceMetadata query (testnet DID): HTTP 200, 748ms. Returned 2 linked resources — "Demo Resource" (id=9ba3922e) and "ResourceName" (id=e733ebb7) — with full metadata including checksum fields.

Content integrity: Computed SHA-256 of the resolved resource bytes and compared against the on-chain checksum field in the metadata. They matched exactly:

e1dbc03b50bdb995961dc8843df6539b79d03bf49787ed6462189ee97d27eaf3

This means the content-addressed integrity guarantee works: you can resolve a resource, hash it yourself, and verify it matches what was anchored on-chain. That is the core security property DLR is built around.

What this means

The addressing layer is live and functional. Sub-second resolution (median ~450ms for DID resolution, ~1s for resource dereference) is fast enough for agent workflows that aren't on a hard real-time budget. The integrity check working end-to-end means DLR's core value proposition — "you can verify this resource was authorized by the DID controller and has not been modified since anchoring" — holds in practice, not just on paper.

What doesn't exist yet: tooling that wires this up to agent credential lifecycle management. The revocation list use case (agent offboarding: publish a status update as a DLR resource, and verifiers check it during presentation) is theoretically supported by this infrastructure but I have not seen it implemented in any agent framework. That's the gap.
