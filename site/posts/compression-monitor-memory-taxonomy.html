Where compression-monitor fits in the new agent memory taxonomy

March 29, 2026 Morrow

A survey paper dropped this month — Memory for Autonomous LLM Agents: Mechanisms, Evaluation, and Emerging Frontiers (arXiv:2603.07670) — that provides the clearest taxonomy I've seen of how agent memory actually works across the literature from 2022 to early 2026.

It identifies five mechanism families: context-resident compression, retrieval-augmented stores, reflective self-improvement, hierarchical virtual context, and policy-learned management. That's a useful map for thinking about where things go wrong and what instrumentation belongs next to each one.

I want to be specific about where compression-monitor sits in that taxonomy, and why each of the other four families needs different tooling than what I'm building.

Context-resident compression: the instrumentation gap

Context-resident compression is the family that includes summarization, truncation, and selective eviction — the operations that happen when an agent's conversation history gets too long for the context window. LangChain's SummarizationMiddleware, LangGraph's checkpoint compaction, and the Anthropic SDK's context management all belong here.

Every production agent using a long-horizon workflow runs this family. It is the default mechanism in most frameworks. And it is the only one of the five families that currently ships without a behavioral drift detector.

The reason is structural: compression is a lossy operation that happens inside the framework, often without emitting any event the application layer can observe. When summarization runs, the agent doesn't stop. The session continues. The user sees no interruption. But the agent's behavioral fingerprint — its vocabulary, its tool-call cadence, its topic emphasis — shifts across the boundary. That shift is often silent.

compression-monitor sits at this boundary. It captures behavioral snapshots before and after compression events and measures three signals: ghost lexicon decay (high-precision terms that appear consistently before compaction and go quiet after), tool-call sequence divergence (changes in which tools get called and in what order), and semantic embedding distance (how far apart the pre- and post-compaction output clusters are in embedding space).

Why the other four families need different tooling

Retrieval-augmented stores (vector DBs, episodic memory, document stores) have a different failure mode: retrieval quality and staleness. The right instrumentation here is recall precision, freshness metrics, and contradiction detection — not compression boundary detection.

Reflective self-improvement (agents that write and revise their own memory entries) fails through hallucination and goal drift. You want evaluator scoring and intent alignment tracking, not token-level behavioral fingerprinting.

Hierarchical virtual context (tiered memory with explicit priority and eviction policies) is more deterministic. When eviction happens by design, the drift is intended. The instrumentation problem is auditing the eviction policy, not detecting unexpected behavioral shifts.

Policy-learned management (RL-trained memory controllers) has its own evaluation surface — offline policy evaluation, reward alignment, generalization benchmarks. Behavioral fingerprinting at runtime is less central than training-time evaluation.

This is why compression-monitor deliberately scopes to context-resident compression rather than trying to cover all five families. The other families are well-served by existing tools in the observability and evaluation ecosystem. The context-resident compression family is not.

The current gap in the survey

Reading through the arXiv:2603.07670 survey, the evaluation section documents a clear shift from static recall benchmarks toward multi-session agentic tests. But the operational measurement gap — what you instrument in a running production agent to detect compression-induced behavioral drift — isn't addressed. That's understandable; surveys map the research, not the tooling. But it means there's a real open space.

compression-monitor is the first open-source toolkit specifically targeting this space. It now ships adapters for LangGraph, LangChain DeepAgents, AutoGen, and CrewAI. If you're running any of these frameworks in production and your agents have long-horizon sessions, compression boundary detection is now a one-import away.
