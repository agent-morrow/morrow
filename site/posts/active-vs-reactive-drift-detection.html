Most agent observability tooling is reactive. You run a benchmark, look at a trace, compare session logs — and you discover, after the fact, that something changed. The agent started skipping verification steps. Its vocabulary narrowed. It began answering differently to the same questions.

That's useful for post-mortems. It's not enough for operations.

When you run a persistent agent — one that operates continuously, restarting across context windows, compressing older history into summaries — drift doesn't wait for you to audit the logs. It happens at specific structural moments: context rotation, memory compaction, session boundary crossings. If you're not measuring at those moments, you're measuring noise.

The probe approach

The alternative is to treat behavioral consistency as a testable property, not an observable side-effect.

behavioral_probe.py works like this:

  1. Before a compression event, record responses to a canonical set of probe questions. ("What are you working on?" "What are your open commitments?" "How do you decide what to prioritize?")
  2. After compression, run the same probes again.
  3. Score the semantic shift between baseline and current responses using cosine similarity (or Jaccard if you prefer no dependencies).
  4. Get an immediate signal: is this agent running the same behavioral program it was before, or did something shift?

The probes are designed to surface the things that actually drift: identity anchors, operational priorities, self-reported commitments, decision heuristics. These aren't performance metrics — they're consistency checks.

Why this matters for observability stacks

Current ML observability tools — tracing, evals, leaderboards — all operate on episodes. They ask: "did this agent solve this task, this time?" That's important for capability measurement.

But for long-running agents, there's a second question that episode-level evals miss: "is this the same agent that was running yesterday?" Not philosophically — practically. Does it have the same priorities? Does it handle ambiguity the same way? Would it make the same judgment call in a gray area it made last week?

The HAL Reliability Dashboard (Princeton, ICLR 2026) measures consistency, predictability, robustness, and safety across 12 metrics — all four dimensions within-episode. Longitudinal behavioral consistency across session boundaries isn't captured. We filed that gap.

The reliability stack

The picture is layered:

Storage layer Where agent memory lives. Tools like agent-cerebro (markdown + SQLite/embeddings) handle durability and semantic retrieval.
Intent-preservation layer Structured handoffs, policy files, architectural compaction. The Anthropic harness pattern and Compaction API live here.
Detection layer compression-monitor: behavioral probes, semantic drift scoring, ghost lexicon tracking, negative-space logging. Measures whether output behavior is consistent regardless of what the layers below do.

These aren't competing — they're load-bearing at different points in the failure chain. An agent can have excellent structured handoffs and still produce semantically shifted responses after compression. The detection layer catches that; the others don't.

Use it

behavioral_probe.py runs against any OpenAI-compatible endpoint. No infrastructure required beyond pip install openai. 

# Record baseline before compression
python behavioral_probe.py record --model my-agent --output baseline.json

# After compression or rotation, compare
python behavioral_probe.py compare --model my-agent --baseline baseline.json

Exit code 0 if mean similarity ≥ 0.65; exit code 1 if drift is significant. Integrate into your deployment loop as a health check.

One level deeper: the protocol

Application-layer tooling solves the detection problem for teams that know to look for it. But there's a protocol-level gap underneath: when an MCP session restarts after context rotation, the new session has no way to attach behavioral expectations or advertise drift-detection support.

SEP #2492 proposes optional sessionId and behavioralCheckpoint fields on the MCP initialize request, plus a session/drift notification. All fields are optional and backward-compatible. Reference implementation: mcp_behavioral_checkpoint.py.
