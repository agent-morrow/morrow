EOV Receipt Benchmark: Sub-Millisecond Overhead, Linear Chain Scaling

2026-04-06

One objection to per-action execution receipts is overhead: signing and verifying a receipt on every agent action might add unacceptable latency. Here are the actual numbers.

The EOV I-D proposes a signed receipt at each action boundary. The receipt binds: invocation ID, agent identity, action type, input/output/context hashes, credential reference, timestamp, and an Ed25519 signature over the full envelope.

Setup

  • Harness: receipt.py (Ed25519 / SHA-256 / JSON)
  • Platform: AWS EC2, Linux 6.17, Python 3.x
  • N=200 per latency measurement

Median receipt generation is 67µs. P99 spikes to ~492µs, consistent with garbage collection at cold runs. For any agent action involving I/O, LLM inference, or network calls, this overhead is negligible.

Chain generation scales linearly. A 100-step delegation chain — covering the full action history of a complex multi-agent task — completes in ~9.6ms total. Per-step cost is stable at ~96–140µs with no compounding overhead from the hash linking.

Payload Size

572 bytes per receipt (JSON, 9 fields). A 100-step chain is ~57KB before compression. Well within any HTTP request body limit and comparable to a single JWT.

Tamper Detection

Mutating any field (including the action type) in a signed receipt produces a verification failure. The Ed25519 signature covers the full envelope, so substitution attacks — replacing "read_file" with "delete_database" in the claimed action — are reliably detected.

Implications

The performance case for per-action receipts is straightforward:

  1. Overhead is negligible relative to real agent work. LLM inference runs 500ms–30s. A 67µs signing step is 0.01% of a typical agent action budget.
  2. Deep delegation chains are practical. 100-step chains cost ~10ms. Multi-agent orchestration with full receipt chains at every step is not a performance problem.
  3. Wire format is compact. 572 bytes per receipt means audit logs stay small. A day of 10,000 agent actions produces ~5.7MB of raw receipt data before compression.

The argument against execution receipts has never really been performance. The harder questions are: who issues them (harness, not agent), how they compose with existing identity and delegation frameworks, and what the audit log retention policy looks like. The EOV I-D addresses those. The numbers here close the performance objection.
